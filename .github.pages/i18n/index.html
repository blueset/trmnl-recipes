i18n.js

A lightweight localization helper for TRMNL recipes. It reads translation strings and a locale from a DOM element, then formats every [data-t] element in the same view using Unicode MessageFormat 2 (via messageformat v4).

URL: https://blueset.github.io/trmnl-recipes/i18n.js

Quick Start

Three steps are all you need to add localization to a recipe layout:

Step 1 — Supply translation strings and locale

Add a container element anywhere in the view that carries the user's locale in its lang attribute and a flat JSON object of translation strings in its data-t-data attribute. There are two ways to supply strings (see Choosing between l_word and inline strings for guidance): 

<!-- Option A: load strings from trmnl-i18n via l_word -->
<div lang="{{ trmnl.user.locale | escape }}"
     data-t-data="{{ 'my_plugin_key' | l_word: trmnl.user.locale | json | escape }}">
  ...
</div>
<!-- Option B: supply strings inline in the template -->
{% assign locales = '{"en":{"greeting":"Hello, {$name}!"},"fr":{"greeting":"Bonjour, {$name}!"}}' | parse_json %}
<div lang="{{ trmnl.user.locale | escape }}"
     data-t-data="{{ locales[trmnl.user.locale] | json | escape }}">
  ...
</div>

Step 2 — Mark translatable elements

Give each element a data-t attribute whose value is the key in the translation dictionary. Put a fallback message (plain text or MessageFormat) between the tags — it is used when no translation is found. 

<span data-t="greeting" data-t-name="Alice">Hello, {$name}!</span>
<span data-t="count" data-t-n="3">{$n} items</span>

Step 3 — Import and call applyPlaintext

{% assign script_id = "my-plugin-" | append_random %}
<script type="module" id="{{ script_id }}" data-t-script>
  import { applyPlaintext } from 'https://blueset.github.io/trmnl-recipes/i18n.js';
  applyPlaintext({}, { scriptId: "{{ script_id }}" });
</script>

That's it — every [data-t] element inside the view is now replaced with the formatted, localized string.

Tutorial: Localizing a Recipe

This tutorial walks through adding full i18n support to a simple plugin that displays a download count. We start with a plain layout and progressively add localization.

Starting point

<!-- download-stats/src/full.liquid -->
<div class="layout layout--col">
  <span class="value--xlarge">{{ data.downloads }}</span>
  <span class="label">downloads</span>
</div>

1. Add translation strings via l_word

Register translation strings for the plugin key download_stats in the trmnl-i18n custom_plugins locale files. For English (en.yml): 

# en.yml
download_stats:
  downloads: ".input {$value :number} .match $value
    zero  {{No downloads}}
    one   {{1 download}}
    *     {{{$value} downloads}}"

Then wrap the layout in a container element that carries the locale and the translated strings: 

<div lang="{{ trmnl.user.locale | escape }}"
     data-t-data="{{ 'download_stats' | l_word: trmnl.user.locale | json | escape }}">
  <div class="layout layout--col">
    <span class="value--xlarge">{{ data.downloads }}</span>
    <span class="label">downloads</span>
  </div>
</div>

2. Mark elements with data-t

Replace the static text with data-t keys and pass the dynamic value via a data-t-value attribute. The original content serves as a fallback when no translation is available. 

<div lang="{{ trmnl.user.locale | escape }}"
     data-t-data="{{ 'download_stats' | l_word: trmnl.user.locale | json | escape }}">
  <div class="layout layout--col">
    <span class="value--xlarge">{{ data.downloads }}</span>
    <span class="label" data-t="downloads" data-t-value="{{ data.downloads }}">
      {% raw %}.input {$value :number} .match $value zero {{No downloads}} one {{1 download}} * {{{$value} downloads}}{% endraw %}
    </span>
  </div>
</div>

Tip: wrapping the fallback content in {% raw %}…{% endraw %} prevents Liquid from accidentally interpreting the MessageFormat curly braces.

3. Add the module script

Generate a unique ID per render to avoid collisions when a view is embedded multiple times, then import and call applyPlaintext: 

{% assign script_id = "download-stats-" | append_random %}
<script type="module" id="{{ script_id }}" data-t-script>
  import { applyPlaintext } from 'https://blueset.github.io/trmnl-recipes/i18n.js';
  applyPlaintext({}, { scriptId: "{{ script_id }}" });
</script>

4. Verify

Open the recipe preview. The label will now show the correct plural form ("1 download", "42 downloads", "No downloads") and will switch language automatically when the user's locale changes.

Bonus: inline strings without l_word

When you supply strings inline, the recipe author is responsible for managing and updating all translations. Use parse_json to hold a locale map in the template and look up the right object by locale at render time: 

{% assign locales = '{"en":{"downloads":".input {$value :number} .match $value zero {{No downloads}} one {{1 download}} * {{{$value} downloads}}"},"fr":{"downloads":".input {$value :number} .match $value zero {{Aucun téléchargement}} one {{1 téléchargement}} * {{{$value} téléchargements}}"}}' | parse_json %}
<div lang="{{ trmnl.user.locale | escape }}"
     data-t-data="{{ locales[trmnl.user.locale] | json | escape }}">
  ...
</div>

Or build / override strings programmatically in JavaScript:

<script type="module" id="{{ script_id }}" data-t-script>
  import { applyPlaintext } from 'https://blueset.github.io/trmnl-recipes/i18n.js';

  // Build / override strings programmatically before applying
  const root = document.getElementById("{{ script_id }}").closest(".view");
  const container = root.querySelector("[data-t-data]");
  const strings = JSON.parse(container.dataset.tData || "{}");
  strings.extra_key = "Custom string: {$value}";
  container.dataset.tData = JSON.stringify(strings);

  applyPlaintext({}, { scriptId: "{{ script_id }}" });
</script>

Documentation

HTML Attributes

lang

Set on the same element as data-t-data. Controls the locale used by MessageFormat and by browser Intl APIs inside custom functions. Typical value: {{ trmnl.user.locale | escape }}.

data-t-data

A JSON-serialized, HTML-escaped flat object mapping translation keys to MessageFormat 2 strings. Must be present on exactly one element inside the .view container that wraps the plugin.

Sources:

  • l_word Liquid filter — looks up strings from the trmnl-i18n custom_plugins locale files and returns the object for a given plugin key and locale. Recipe authors and translators contribute via pull requests to trmnl-i18n, enabling public and centralized translation. 
    data-t-data="{{ 'my_plugin' | l_word: trmnl.user.locale | json | escape }}"
  • Inline Liquid strings — define a locale map directly in the template using parse_json; the recipe author is responsible for maintaining all translations. 
    {% assign locales = '{"en":{"hello":"Hello, {$name}!"},"fr":{"hello":"Bonjour, {$name}!"}}' | parse_json %}
data-t-data="{{ locales[trmnl.user.locale] | json | escape }}"
  • JavaScript — set element.dataset.tData before calling applyPlaintext/applyHTML.

data-t

Translation key for the element. The library looks up this key in the dictionary supplied by data-t-data. If the key is not found, the element's existing innerHTML is used as the MessageFormat pattern instead (fallback). 

<span data-t="greeting">Hello!</span>

data-t-variable

Passes a named variable to the MessageFormat formatter. The attribute name after the data-t- prefix is converted to camelCase and becomes the variable name accessible as {$variable} in the pattern.

HTML attributeMessageFormat variable
data-t-value="42"{$value}
data-t-next-days="3"{$nextDays}
data-t-author-name="Alice"{$authorName}

Values that are valid JSON (numbers, booleans, arrays, objects) are parsed automatically; all other values are treated as plain strings.

data-t-script

Marks the <script type="module"> element that imports and calls this library. The library uses it to locate the parent .view container so that multiple plugin instances on the same page stay isolated from each other.

When there is only one plugin instance on a page you can omit id and scriptId, and the library will fall back to document.querySelector("script[data-t-script]"). For safety, always generate a unique ID per render: 

{% assign script_id = "my-plugin-" | append_random %}
<script type="module" id="{{ script_id }}" data-t-script>
  import { applyPlaintext } from 'https://blueset.github.io/trmnl-recipes/i18n.js';
  applyPlaintext({}, { scriptId: "{{ script_id }}" });
</script>

JavaScript API

applyPlaintext(functions?, options?)

Finds every [data-t] element in the view, formats its translation string, and sets element.innerText to the result. Use this for elements that should contain only plain text.

ParameterTypeDescription
functions object Optional map of custom MessageFormat functions (see Custom Functions). Merged on top of DraftFunctions.
options.scriptId string The id of the data-t-script element. Required when multiple plugin instances are on the same page. 
import { applyPlaintext } from 'https://blueset.github.io/trmnl-recipes/i18n.js';
applyPlaintext({}, { scriptId: "{{ script_id }}" });

applyHTML(functions?, markups?, options?)

Same as applyPlaintext, but uses formatToParts so that markup tags in the translation pattern are converted to real DOM nodes. Use this when translations need to produce styled HTML output.

ParameterTypeDescription
functions object Same as for applyPlaintext.
markups object Map from markup tag name to a factory function (options, children: Node[]) => Node (see Custom Markup).
options.scriptId string Same as for applyPlaintext. 
import { applyHTML } from 'https://blueset.github.io/trmnl-recipes/i18n.js';
applyHTML(
  { /* custom functions */ },
  {
    bold: (options, children) => {
      const b = document.createElement('b');
      children.forEach(c => b.appendChild(c));
      return b;
    },
    br: () => document.createElement('br'),
  },
  { scriptId: "{{ script_id }}" }
);

DefaultFunctions, DraftFunctions

Re-exports from messageformat@4. Useful when writing custom functions that delegate back to the built-in formatter: 

import { DefaultFunctions } from 'https://blueset.github.io/trmnl-recipes/i18n.js';
// or import directly from messageformat:
import { DefaultFunctions } from 'https://cdn.jsdelivr.net/npm/messageformat@4.0.0-13/lib/functions/+esm';

MessageFormat Syntax

Translation strings follow the Unicode MessageFormat 2 specification (implemented by messageformat v4). See also the MessageFormat 2 tutorial. Below are the patterns used most often in TRMNL recipes.

Variable interpolation

Hello, {$name}!

Inserts the value of data-t-name.

Number formatting

Population: {$count :number}

Formats the number according to the locale (thousands separators, etc.). Requires data-t-count="1234567".

Plural selection

.input {$value :number}
.match $value
  zero  {{No items}}
  one   {{1 item}}
  *     {{{$value} items}}

The .input declaration annotates $value as a number so that plural categories (zero, one, two, few, many, *) work correctly for the current locale. * is the default / catch-all branch.

Markup tags (applyHTML only)

{#bold}important{/bold} text and a{#br/}line break
  • {#tagName}…{/tagName} — wrapping markup; maps to a DOM node via the markups object.
  • {#tagName/} — standalone (self-closing) markup.
  • Unknown tag names are silently ignored.

Escaping Liquid curly braces

Liquid interprets { and }, so wrap MessageFormat patterns in {% raw %}…{% endraw %} when they appear as inline HTML content: 

<span data-t="count" data-t-value="{{ data.n }}">
  {% raw %}.input {$value :number} .match $value one {{1 item}} * {{{$value} items}}{% endraw %}
</span>

The l_word Liquid Filter and Choosing a Strings Source

How l_word works

l_word is a TRMNL platform Liquid filter that loads a localized string dictionary for a given plugin key and locale. 

{{ 'plugin_key' | l_word: trmnl.user.locale | json | escape }}
StepWhat it does
l_word: trmnl.user.localeReturns the Ruby hash / object for plugin_key in the user's locale, falling back to English.
jsonSerializes the hash to a JSON string.
escapeHTML-escapes the JSON string so it is safe to embed in an attribute value.

Strings are defined in the trmnl-i18n repository under lib/trmnl/i18n/locales/custom_plugins/. Each file is a YAML file keyed by locale code (e.g., en.yml, ja.yml). The top-level key must match the argument you pass to l_word: 

# en.yml
my_plugin:
  greeting: "Hello, {$name}!"
  item_count: ".input {$n :number} .match $n one {{1 item}} * {{{$n} items}}"

If no translation exists for the user's locale, the platform falls back to English. If the key is missing entirely, data-t-data will be an empty object ({}) and the elements' original HTML content will be used as-is.

Choosing between l_word and inline strings

l_wordInline strings
Who manages translations Anyone — contributors send pull requests to trmnl-i18n, enabling public and centralized translation. The recipe author is solely responsible for adding and maintaining all locale strings.
Adding a new language Open a PR in trmnl-i18n; no change to the recipe itself. Edit the recipe template to add the new locale.
When to prefer Recipes intended for broad, multilingual audiences. Quick prototypes, single-language recipes, or strings tightly coupled to recipe logic.

Custom Functions

Both applyPlaintext and applyHTML accept an optional functions object. Each entry is a MessageFormat 2 function that can be applied to a value with {$variable :functionName}.

A custom function has the signature:

functionName(context, options, input) => MessageValue
ArgumentDescription
contextMessageFormat context object; context.locales is the array of active locale strings.
optionsOptions passed in the pattern, e.g. {$v :fn opt=val}.
inputThe resolved value of the variable.

Return a value produced by one of the built-in functions (e.g., DefaultFunctions.string(context, options, result)) or any messageformat-compatible MessageValue object.

Example — a :langName function that converts a BCP 47 tag to a human-readable language name (from the Tatoeba recipe): 

import { applyPlaintext, DefaultFunctions } from 'https://blueset.github.io/trmnl-recipes/i18n.js';

applyPlaintext({
  langName: (context, options, input) => {
    try {
      const dn = new Intl.DisplayNames(context.locales, { type: "language" });
      const name = dn.of(input) ?? input;
      return DefaultFunctions.string(context, options, name);
    } catch {
      return DefaultFunctions.string(context, options, input);
    }
  },
}, { scriptId: "{{ script_id }}" });

Usage in the translation pattern:

Translating from {$from :langName}

Custom Markup (applyHTML)

When using applyHTML, translation strings may contain markup tags ({#tagName}…{/tagName}). Each tag name must be mapped to a factory function in the markups object: 

markupName: (options, children) => Node
ArgumentDescription
optionsKey/value options from the pattern, e.g. {#tag key=val}.
childrenArray of Node objects produced by the nested content.

The function must return a single DOM Node.

Example — rich layout with styled spans and line breaks (from the Custom Next Holiday recipe): 

import { applyHTML } from 'https://blueset.github.io/trmnl-recipes/i18n.js';

applyHTML({}, {
  mainValue: (options, children) => {
    const node = document.createElement('span');
    node.className = "value--xxlarge";
    children.forEach(c => node.appendChild(c));
    return node;
  },
  mainLabel: (options, children) => {
    const node = document.createElement('span');
    node.className = "value--base";
    children.forEach(c => node.appendChild(c));
    return node;
  },
  br: () => document.createElement('br'),
}, { scriptId: "{{ script_id }}" });

Translation pattern using the markup tags:

{#mainLabel}Today is{/mainLabel}{#br/}{#mainValue}{$name}{/mainValue}

Tags with no matching entry in markups are silently skipped; their text content is still rendered. Unclosed tags are automatically closed at the end of the message.
